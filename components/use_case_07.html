Detail of proposed monitoring system components for use case 7

Use case description: Receive notifications of dataplane connectivity failure between GENI sites, including sites which are totally unreachable, shared interfaces which are down, interfaces whose utilization is too high, and experimental VLANs which can't be used successfully. Share results of dataplane connectivity testing among operators to use the network efficiently.

Use case implementation story: The first source of data for this use case is time-series traffic metrics and VLAN membership data for shared interfaces of rack and core network devices. Each rack and core network will collect these metrics for themselves, and store them in their own local datastore. The second type of data is the results of end-to-end network traffic checks using experimental networks. This use case requires a mechanism for testing health across a dynamic dataplane connection — this mechanism will probably be a framework by which an external checker orchestrates pings between VMs in GENI slivers across links of interest. Health check metrics about success/failure of those connections will be pushed or pulled from those slivers to a local datastore. Then each group that wants to send notifications about dataplane network health will run an aggregator that polls the local datastores for the active dataplane testing and for any networks or racks they want to monitor, and the aggregator will share recent values with an alerting service, which will detect out-of-bounds values at that group's preference and send alerts. Any groups that want to produce visualization of traffic will build this on top of their aggregator as well.
